PHASE 3: RESPONSIVENESS RULES
====================================================================
In /assets/css/style.css, implement mobile-first responsive design using CSS media queries at these breakpoints:
- Base styles: mobile (< 640px) — single column layouts, stacked nav collapsed into a hamburger menu, full-width notice cards
- @media (min-width: 640px): tablet — 2-column notice grid, expanded nav
- @media (min-width: 1024px): desktop — 3-column notice grid for public view, sidebar + main content layout for admin dashboard
- @media (min-width: 1280px): large desktop / kiosk display — larger font sizes for the kiosk.php auto-cycling display mode, since it's meant to be viewed from a distance on a mounted screen

Ensure all buttons and tappable elements have a minimum touch target of 44px height for mobile usability. Use CSS Grid for notice card layouts and Flexbox for nav/header/footer components. Add a hamburger menu toggle in /assets/js/main.js that shows/hides mobile nav via a class toggle (no framework dependency).

PHASE 4: DATABASE SCHEMA (PostgreSQL)
====================================================================
Create /sql/schema.sql defining:

TABLE users
- id SERIAL PRIMARY KEY
- name VARCHAR(150) NOT NULL
- email VARCHAR(150) UNIQUE NOT NULL
- password_hash VARCHAR(255) NOT NULL
- role VARCHAR(20) NOT NULL DEFAULT 'viewer' -- 'super_admin', 'admin', 'viewer'
- created_at TIMESTAMP DEFAULT NOW()

TABLE categories
- id SERIAL PRIMARY KEY
- name VARCHAR(100) NOT NULL
- description TEXT

TABLE notices
- id SERIAL PRIMARY KEY
- title VARCHAR(255) NOT NULL
- body TEXT NOT NULL
- category_id INTEGER REFERENCES categories(id) ON DELETE SET NULL
- posted_by INTEGER REFERENCES users(id) ON DELETE SET NULL
- priority VARCHAR(20) DEFAULT 'normal' -- 'normal', 'urgent'
- status VARCHAR(20) DEFAULT 'draft' -- 'draft', 'published', 'archived'
- publish_at TIMESTAMP
- expires_at TIMESTAMP
- created_at TIMESTAMP DEFAULT NOW()
- updated_at TIMESTAMP DEFAULT NOW()

TABLE attachments
- id SERIAL PRIMARY KEY
- notice_id INTEGER REFERENCES notices(id) ON DELETE CASCADE
- file_path VARCHAR(255) NOT NULL
- file_type VARCHAR(50)
- uploaded_at TIMESTAMP DEFAULT NOW()

TABLE activity_logs
- id SERIAL PRIMARY KEY
- admin_id INTEGER REFERENCES users(id) ON DELETE SET NULL
- action VARCHAR(100) NOT NULL -- 'created', 'edited', 'deleted', 'published', 'archived'
- notice_id INTEGER REFERENCES notices(id) ON DELETE SET NULL
- details TEXT
- timestamp TIMESTAMP DEFAULT NOW()

Add indexes: CREATE INDEX idx_notices_status ON notices(status); CREATE INDEX idx_notices_publish_at ON notices(publish_at);

Create /sql/seed.sql with sample data: 1 super_admin user, 2 admin users, 3 viewer users, 5 categories (Academic, Administrative, Events, Exams, General), and 8 sample notices with varying statuses/priorities for testing.

PHASE 7: BACKEND — CONTROLLERS & ENDPOINTS
====================================================================
Build controllers in /app/controllers handling both full-page rendering (for initial loads) and JSON responses (for AJAX calls). Organize routes through Router.php.

1. AuthController.php
   - GET /login → render login view
   - POST /login → validate credentials, start session, redirect by role
   - POST /logout → destroy session, redirect to login
   - GET /register (viewers only, if self-registration is allowed) → render + handle registration with password hashing (password_hash())

2. NoticeController.php
   - GET /admin/notices → list all notices (admin, with filters: status, category)
   - GET /admin/notices/create → render notice form
   - POST /admin/notices/create → validate + insert notice, handle file upload via move_uploaded_file() with type/size validation (allow pdf, jpg, png, max 5MB), log action in activity_logs
   - GET /admin/notices/edit/{id} → render edit form pre-filled
   - POST /admin/notices/edit/{id} → update notice, log action
   - POST /admin/notices/delete/{id} → soft-delete or hard-delete, remove attachment file, log action
   - GET /api/notices/active → JSON endpoint returning active published notices (used by AJAX polling on public pages and kiosk display)
   - GET /api/notices/search?q= → JSON endpoint for keyword search

3. CategoryController.php
   - CRUD endpoints for categories (admin only)

4. UserController.php
   - GET /admin/users → list users (super_admin only)
   - POST /admin/users/role/{id} → update user role (super_admin only)
   - POST /admin/users/delete/{id} → delete user

5. DashboardController.php
   - GET /admin/dashboard → render stats: total notices, active vs expired count, most-viewed notices, recent activity log feed

6. PublicController.php
   - GET / → render home.php showing active notices in a responsive grid, with category filter and search bar
   - GET /notice/{id} → render notice-detail.php with full content + attachment download link
   - GET /kiosk → render kiosk.php, a full-screen auto-cycling display of active notices (JS auto-rotates every N seconds), designed for a physically mounted screen

Every state-changing endpoint (POST/DELETE-style actions) must validate the CSRF token from Auth.php before processing.
